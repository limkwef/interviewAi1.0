# 数据库设计

## 1. 设计原则

| 原则 | 说明 |
|---|---|
| **JSON 优先** | AI 解析的简历数据整段存入 JSON 字段，数据库不关心内部结构 |
| **索引按需添加** | 只有被 WHERE 子句高频查询的字段才单独建列+索引 |
| **版本只增不删** | 每次上传/填写都新增记录，is_active 标记当前生效版本 |
| **最小冗余** | 仅冗余高频查询字段（如 user_id、skill_name）避免 Join，其他不拆 |

## 2. DDL

### 2.1 简历主表

```sql
CREATE TABLE `resume` (
    `id`            BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
    `user_id`       BIGINT       NOT NULL                COMMENT '用户ID',
    `version`       INT          NOT NULL DEFAULT 1      COMMENT '版本号，同用户下递增',
    `source`        VARCHAR(20)  NOT NULL DEFAULT 'upload' COMMENT '来源：upload-上传解析, template-在线填写',
    `status`        TINYINT      NOT NULL DEFAULT 0      COMMENT '解析状态：0-解析中 1-解析完成 2-解析失败',
    `error_msg`     VARCHAR(500) DEFAULT NULL             COMMENT '解析失败原因',
    `raw_text`      MEDIUMTEXT   DEFAULT NULL             COMMENT '原始提取文本（PDF/Word 提取的纯文本）',
    `parsed_data`   JSON         DEFAULT NULL             COMMENT 'AI 解析结果 / 表单提交的结构化数据',
    `file_name`     VARCHAR(255) DEFAULT NULL             COMMENT '原始文件名（仅上传来源）',
    `file_url`      VARCHAR(500) DEFAULT NULL             COMMENT '文件存储路径（仅上传来源）',
    `file_size`     BIGINT       DEFAULT NULL             COMMENT '文件大小（字节）',
    `is_active`     TINYINT      NOT NULL DEFAULT 0       COMMENT '是否当前生效：0-否 1-是',
    `create_time`   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '修改时间',

    PRIMARY KEY (`id`),
    INDEX `idx_user_id` (`user_id`),
    INDEX `idx_user_active` (`user_id`, `is_active`),
    INDEX `idx_create_time` (`create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='简历主表';
```

### 2.2 面试-简历关联表（可选）

```sql
CREATE TABLE `resume_session` (
    `id`          BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键',
    `resume_id`   BIGINT NOT NULL COMMENT '简历ID',
    `session_id`  BIGINT NOT NULL COMMENT '面试会话ID',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_session` (`session_id`),
    INDEX `idx_resume_id` (`resume_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='面试-简历关联表';
```

> 如果面试绑定简历是高频路径，考虑在 `interview_session` 表加 `resume_id` 字段而非独立关联表，减少一次 Join。

## 3. 索引使用场景一览

| SQL 模式 | 命中索引 |
|---|---|
| `WHERE user_id = ? AND is_active = 1` | `idx_user_active` |
| `WHERE user_id = ? ORDER BY version DESC` | `idx_user_id`（filesort） |
| `WHERE resume_id = ?`（关联表） | `idx_resume_id` |
| `WHERE session_id = ?`（关联表） | `uk_session` |

## 4. JSON 字段查询策略

虽然 `parsed_data` 是 JSON 字段，MySQL 8.0+ 提供 JSON 函数，但**拒绝在业务 SQL 中写 JSON_EXTRACT**——需要查询的字段应当冗余为独立列。

仅在以下场景使用 JSON 函数：
- 管理后台的全量统计（慢可接受，如"统计所有简历中最常见的 10 个技能"）
- 数据迁移/清洗脚本

```sql
-- 可接受的运维查询（不阻塞业务路径）
SELECT 
    JSON_UNQUOTE(JSON_EXTRACT(parsed_data, '$.skills[*].name')) AS skills
FROM resume 
WHERE is_active = 1;
```

## 5. 版本控制策略

```
同一条简历修改后不会 UPDATE，而是 INSERT 新记录：

version=1 (source=upload,  is_active=0)  ← 第一次上传
version=2 (source=template, is_active=1) ← 在解析结果上编辑保存，来源标记为 template
version=3 (source=upload,  is_active=0)  ← 重新上传新版本
version=4 (source=upload,  is_active=1)  ← 最新版激活，自动把旧的 is_active=0
```

当 `is_active` 从 0→1 时，由应用层保证同 user_id 下只有一个 is_active=1。

## 6. 与现有 User 表的边界

| user 表字段 | 是否与简历同步 | 原因 |
|---|---|---|
| `targetPosition` | ❌ 不同步 | user 字段是用户注册时手填的简短信息，简历的 targetPosition 是解析/填写所得，互不覆盖 |
| `techStack` | ❌ 不同步 | user 字段是逗号分隔的简短标签，简历 skills 是结构化数组，精度不同 |

> 在设计评审时必须坚持：**User 是身份主体，Resume 是业务数据，二者不交叉写入**。如果未来有"从简历同步到 user"的需求，单独建同步任务，不走业务代码。

```sql
-- 技能索引表（性能优化，初期不需要建）

CREATE TABLE `resume_skill_index` (

    `skill_level` VARCHAR(20) DEFAULT NULL,

    INDEX `idx_user_id` (`user_id`)

```

```sql
-- 简历-面试-诊断关联视图（方便运营拉取数据）
CREATE VIEW `v_resume_interview_diagnosis` AS
SELECT 
    r.id AS resume_id, r.user_id, r.version, r.source,
    rs.session_id,
    ir.total_score, ir.level
FROM resume r
LEFT JOIN resume_session rs ON rs.resume_id = r.id
LEFT JOIN interview_report ir ON ir.session_id = rs.session_id;
```
