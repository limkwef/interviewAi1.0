# 需求分析与用户故事

## 1. 业务背景

当前 AI 模拟面试系统使用通用题库（按「方向+难度」出题），无法感知用户的具体技能栈和项目经历，导致：
- 面试问题与用户实际能力脱节
- 诊断报告缺少个性化基线
- 无法评估简历声称水平与真实表现的差距

简历模块的目标是**让 AI 面试官先读简历，再出题**。

## 2. 功能需求清单

### P0 — 核心链路（必须）

| ID | 需求 | 描述 | 优先级 |
|---|---|---|---|
| R-01 | 简历上传解析 | 支持 PDF/Word/图片上传，调用 AI 提取为结构化数据 | P0 |
| R-02 | 简历在线填写 | 提供结构化表单让用户逐项填写简历信息 | P0 |
| R-03 | 简历激活选择 | 用户可在多份简历中选一个作为当前面试简历 | P0 |
| R-04 | 简历绑定面试 | 面试创建时可选绑定简历，AI 据此定制出题 | P0 |

### P1 — 体验增强（重要）

| ID | 需求 | 描述 | 优先级 |
|---|---|---|---|
| R-05 | 上传后编辑 | 上传解析后允许用户在表单中修改 AI 提取结果 | P1 |
| R-06 | 版本管理 | 保留用户所有历史简历，支持版本回退 | P1 |
| R-07 | 多简历管理 | 列表展示所有简历，标注活跃状态和来源 | P1 |

### P2 — 高级能力（加分）

| ID | 需求 | 描述 | 优先级 |
|---|---|---|---|
| R-08 | 技能验证报告 | 面试后输出「简历声称 vs 实际表现」对比 | P2 |
| R-09 | 简历版本对比 | 对比两版简历的技能/项目变化，形成成长轨迹 | P2 |
| R-10 | 简历优化建议 | 基于面试表现和行业数据，建议简历上应补充/突出的技能 | P2 |

## 3. 用户故事

### 3.1 应届生小李

> "我马上要面试了，但简历还没做。我想直接在系统里填一下基本信息和技术栈，让 AI 模拟面试官按照我写的内容来问我。"

**对应方案**：在线填写 → 自动激活 → 创建面试时绑定

### 3.2 在职开发者小王

> "我有一份 PDF 简历，想直接上传。但 AI 提取的可能不准，我需要能改。"

**对应方案**：上传解析 → 进入编辑页确认/修改 → 保存激活

### 3.3 跳槽达人小张

> "我上半年有一份简历，最近更新了。我想看看两份简历之间 AI 验证出我哪些技术真的进步了。"

**对应方案**：多版本管理 + 版本对比功能

### 3.4 面试官视角（产品运营）

> "我想知道平台上用户的技能分布，哪些技术被标注最多，哪些技能缺口最大。"

**对应方案**：从 `parsed_data` 中提取技能做聚合统计（管理后台）

## 4. 边界场景

| 场景 | 处理策略 |
|---|---|
| 上传空文件 / 非 PDF/Word/图片 | 前端预校验 + 后端拒绝并返回错误原因 |
| PDF 是扫描件（无文字层） | 提示用户"扫描件暂不支持，请上传可复制文字的 PDF"，无 OCR 需求时返回友好错误 |
| 简历内容过长（>5000 词） | 截断到 LLM token 上限前 8000 token，提示用户 |
| AI 解析返回不完整 JSON | AIService 的 chatStructured 有重试机制，3 次仍失败→status=2 并记录 error_msg |
| 用户上传了别人的简历 | 系统无法识别，AI 会按照提取到的信息出题——视为用户行为 |
| 面试中更新了简历 | 当前进行中的面试仍使用创建时绑定的版本；新面试使用最新 is_active 简历 |
| 删除活跃简历 | 拒绝删除，提示用户先激活另一份 |
| 同一技能在解析中重复出现（项目经验和技能标签都有） | AI 在 deduplication prompt 中已做去重处理，项目来源标记 `project`，标签来源标记 `skill` |
| 上传简历后调用的 AI 服务超时/不可用 | 走 circuit breaker 降级，status=2，提示用户稍后重试或改用在线填写 |
