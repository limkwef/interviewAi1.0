# 检验标准与验收规范

## 1. 验收层级

验收分为三个层级，逐级提升标准，低层级通过后方可进入下一级：

```
L1: 功能正确性  ────  能用，不报错
      ↓
L2: 工程质量    ────  规范、可维护、可扩展
      ↓
L3: 非功能指标  ────  性能、安全、兼容性
```

---

## 2. L1 — 功能验收

### 2.1 上传解析（source=upload）

| 编号 | 用例 | 预期 | 方式 |
|---|---|---|---|
| TC-U01 | 上传一份标准 PDF 简历（含教育、工作、技能） | 状态码 200，status=0，3-10s 后 status=1，parsed_data 非空 | 手动 + 接口 |
| TC-U02 | 上传纯文本 TXT 简历 | 同 TC-U01 | 手动 |
| TC-U03 | 上传空文件 | 返回 400，errorMsg 非空 | 接口 |
| TC-U04 | 上传 10MB+ 的文件 | 返回 413 或被 Spring 拒绝 | 手动 |
| TC-U05 | 上传不支持的文件类型（.exe .png 等） | 返回 400，提示文件类型不支持 | 接口 |
| TC-U06 | 上传的 PDF 是扫描件（无文字层） | 返回 status=2，error_msg="未能从文件中提取到文本" | 手动 |
| TC-U07 | 上传后立即查询 status，轮询直到完成 | status 从 0→1 或 0→2 | 接口 |
| TC-U08 | 上传成功后的简历出现在列表页 | list 接口返回该记录 | 接口 |
| TC-U09 | AI 解析返回的内容与简历原文在技能、项目上大致匹配 | 人工对比 5 份简历的解析结果，技能召回率 > 80% | 手动 |
| TC-U10 | AI 服务不可用时上传 | 返回 500，错误提示清晰 | 接口 |

### 2.2 在线填写（source=template）

| 编号 | 用例 | 预期 | 方式 |
|---|---|---|---|
| TC-F01 | 填写所有必填字段提交 | 状态码 200，status=1 | 接口 |
| TC-F02 | 只填写部分字段（如只填 basicInfo + skills） | 成功，缺失字段为 null | 接口 |
| TC-F03 | 提交空的 JSON 对象 `{}` | 返回 400，提示至少填写基本信息 | 接口 |
| TC-F04 | skills.level 传入非法值（如"牛逼"） | 后端自动降级为"熟悉"，不抛错 | 接口 |
| TC-F05 | 填写后自动成为 is_active | 新创建的简历 is_active=1 | 接口 |

### 2.3 简历管理

| 编号 | 用例 | 预期 |
|---|---|---|
| TC-M01 | 列表按 create_time DESC 排序 | 最新的在前面 |
| TC-M02 | 分页正常工作 | page/size 参数生效 |
| TC-M03 | A 用户看不到 B 用户的简历 | 跨用户查询返回 404 |
| TC-M04 | 激活简历 B 后，原活跃简历 A 变为 is_active=0 | 二者不会同时活跃 |
| TC-M05 | 同一用户只能有一条 is_active=1 | 数据库层不约束，应用层保证 |
| TC-M06 | 删除 is_active=1 的简历 | 拒绝，返回 400 |
| TC-M07 | 删除已绑定面试的简历 | 拒绝，返回 400 |
| TC-M08 | 删除不存在的 ID | 返回 404 |

### 2.4 面试绑定

| 编号 | 用例 | 预期 |
|---|---|---|
| TC-B01 | 创建面试时携带简历 ID | 面试开始后 AI 的回答引用简历技能 |
| TC-B02 | 创建面试时不携带简历 ID | 行为不变（向后兼容） |
| TC-B03 | 面试进行中切换活跃简历 | 当前面试不受影响（使用创建时的快照） |
| TC-B04 | 绑定已删除的简历 | 返回 400 |

### 2.5 编辑（上传后修改）

| 编号 | 用例 | 预期 |
|---|---|---|
| TC-E01 | 上传解析后编辑保存 | 新版本 version+1，source=template |
| TC-E02 | 编辑后原版本保留 | 列表可见旧版本 |
| TC-E03 | 编辑后新版本自动激活 | is_active=1，旧版变为 0 |

### 2.6 版本对比

| 编号 | 用例 | 预期 |
|---|---|---|
| TC-C01 | 对比两个版本，技能有增删 | 报告中正确展示新增/移除/升降级 |
| TC-C02 | 对比两个版本，内容完全一致 | 报告为空差异 |
| TC-C03 | 对比不存在的版本 ID | 返回 404 |

### 2.7 技能验证报告

| 编号 | 用例 | 预期 |
|---|---|---|
| TC-V01 | 面试结束后验证报告生成 | 报告中有 skillResults 数组 |
| TC-V02 | 简历声称"精通"且面试回答好的技能 | verifiedLevel=精通，gap=0 |
| TC-V03 | 简历声称"精通"但面试表现差的技能 | verifiedLevel<精通，gap<0 |
| TC-V04 | 面试中未涉及简历中的技能 | 该技能不出现在报告中 |

## 3. L2 — 工程规范性

### 3.1 Java 后端规范

| 检查项 | 标准 | 验证方式 |
|---|---|---|
| 包结构 | `controller/` `service/` `entity/` `mapper/` 各归其位 | Code Review |
| 命名风格 | `ResumeController`、`ResumeService`、`ResumeMapper` 驼峰命名 | Code Review |
| Controller 厚度 | Controller 不做业务判断，仅参数校验 + 调用 Service | Code Review |
| 事务标注 | 激活切换、删除等写操作标 `@Transactional` | Code Review |
| 异常分层 | 业务异常抛 `BusinessException(code, msg)`，非 `RuntimeException` | Code Review |
| 错误码枚举 | 使用统一错误码（如 `ResumeErrorCode.FILE_TYPE_NOT_SUPPORTED`） | Code Review |
| 日志 | Service 层关键节点打 `log.info`/`log.warn`/`log.error` | Code Review |
| MyBatis 映射 | Mapper XML 中 SQL 写 `resume` 表，不跨表 JOIN（关联查询走 Service） | Code Review |
| 单元测试 | Service 层核心方法覆盖率 > 80% | `mvn test` |
| JSON 序列化 | `parsed_data` 使用 JacksonTypeHandler，不用手动 `ObjectMapper` | Code Review |

### 3.2 Vue 前端规范

| 检查项 | 标准 | 验证方式 |
|---|---|---|
| 目录结构 | `views/resume/` `api/resume.js` `router/index.js` 新增路由 | Code Review |
| 组件命名 | `ResumeList.vue` `ResumeForm.vue` 大驼峰 | Code Review |
| 状态管理 | 面试绑定相关的状态放 Pinia store，简历列表自身状态放组件内 | Code Review |
| API 封装 | `api/resume.js` 统一管理 axios 调用 | Code Review |
| 加载态 | 上传/保存/列表都有 loading 状态 | 手动测试 |
| 错误态 | 接口失败时有 Toast 提示 | 手动测试 |
| 空态 | 列表为空时显示"暂无简历" | 手动测试 |
| 表单校验 | 必填字段*标记 + 提交前校验 | 手动测试 |
| 响应式 | 768px-1920px 宽度正常显示 | 手动测试 |

### 3.3 数据库规范

| 检查项 | 标准 | 验证方式 |
|---|---|---|
| 字段注释 | 所有字段有 `COMMENT` | `SHOW FULL COLUMNS` |
| 字符集 | `utf8mb4` + `utf8mb4_unicode_ci` | DDL 检查 |
| 索引命名 | `idx_字段名` 或 `uk_字段名` | DDL 检查 |
| JSON 字段 | `parsed_data` 用 `JSON` 类型，不用 `TEXT` + 自行序列化 | DDL 检查 |
| 数据删除 | 不物理删除（除非用户主动触发删除） | Code Review |

### 3.4 接口规范

| 检查项 | 标准 | 验证方式 |
|---|---|---|
| RESTful | 资源路径使用名词复数 `/api/resume/` | Code Review |
| HTTP 方法 | 查询用 GET、创建 POST、修改 PUT、删除 DELETE | Code Review |
| 统一响应 | 全部返回 `Result` 封装 | Code Review |
| 分页格式 | `{ records: [], total, page, size }` | 接口测试 |
| 错误格式 | `{ code, message }` | 接口测试 |

## 4. L3 — 非功能指标

### 4.1 性能指标

| 指标 | 阈值 | 测量工具 | 备注 |
|---|---|---|---|
| PDF 解析耗时（不含 AI） | < 500ms | 日志 + 压测 | PDFBox 提取速度 |
| AI 解析耗时（P95） | < 10s | 日志聚合 | 包含 DeepSeek API 调用 |
| 上传接口吞吐 | > 5 req/s | JMeter / wrk | 受限于 AI API 并发 |
| 简历列表页接口 P95 | < 200ms | 日志聚合 | 分页查询 |
| 简历详情接口 P95 | < 100ms | 日志聚合 | 主键查询 |
| 并发上传（同一用户） | 排队处理，不超过 1 个 | 代码逻辑 | 同一用户同时上传时，后一个等待前一个解析完成 |

### 4.2 安全指标

| 检查项 | 标准 | 验证方式 |
|---|---|---|
| 文件上传类型白名单 | 仅允许 `.pdf .docx .txt` | 手工测试 + 代码审查 |
| 文件大小限制 | 10MB | `multipart.max-file-size` |
| 文件存储路径 | 不暴露原始路径到前端返回中 | 代码审查 |
| 跨用户越权 | A 用户无法操作 B 用户简历 | 手工测试（改 ID 调用接口） |
| XSS | 解析结果中的技能名称/项目名渲染前转义 | 代码审查 |
| JWT 校验 | 所有 /api/resume/* 受 UserInterceptor 保护 | 代码审查 |
| 文件后缀校验 | 校验 Content-Type + 扩展名双重白名单 | 代码审查 |

### 4.3 兼容性清单

| 项目 | 要求 |
|---|---|
| 浏览器 | Chrome 100+、Edge 100+、Firefox 115+ |
| PDF 版本 | PDF 1.3 - 2.0（PDF/A 兼容） |
| 文件编码 | UTF-8、GBK（中文简历常见编码） |
| 现有接口 | 所有原有 API 签名不变，新增参数 `resumeId` 可选 |
| 现有页面 | 面试配置页 UI 增加简历选择器，不改变现有布局 |

## 5. 冒烟测试清单

以下为上线前必须通过的核心用例，共 10 条，全部通过视为冒烟合格：

```
□ 1. 上传 PDF 简历 → 解析成功 → 数据在列表可见
□ 2. 在线填写简历 → 保存成功 → 数据在列表可见
□ 3. 激活简历 B → A 自动变为非活跃
□ 4. 创建面试绑定简历 → AI 问题与简历技能相关
□ 5. 删除非活跃简历 → 成功
□ 6. 删除活跃简历 → 被拒绝
□ 7. 上传不支持的文件类型 → 友好错误提示
□ 8. A 用户通过 URL 修改 B 用户简历 → 被拒绝
□ 9. 列表分页正常
□ 10. 原有面试功能（不绑简历）不受影响
```
