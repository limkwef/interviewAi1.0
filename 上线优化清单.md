## 二、优点（相比同类项目）

### 1. 功能完整度高
- 多轮追问（maxFollowUp 限制）— 大部分开源面试系统没有追问能力
- 语音交互（百度 TTS/ASR）— 极少有同类项目集成语音
- 简历解析 → 个性化面试（PDFBox + AI 解析）— 差异化功能
- AI 深度诊断报告（维度评分 + 竞品分析 + 改进建议）
- 错题本 + 错题重做 — 学习闭环
- 学习路径推荐 — 基于诊断结果
- 管理后台（用户/题目/日志/反馈）— 可运营

### 2. 技术选型合理
- SSE 流式输出比 WebSocket 轻量，适合单向 AI 推流场景
- Redis 缓存面试上下文避免每轮对话查库
- application-level 随机抽题替代 ORDER BY RAND() — 已优化
- JWT + tokenVersion 支持踢下线，比纯 JWT 更安全

### 3. 架构分层清晰
- Controller → Service → Mapper 三层分明
- 面试模块已拆分为 InterviewService + QuestionSelector + InterviewEvaluateService
- CacheService 封装 Redis 操作，统一 SCAN 替代 KEYS
- PromptBuilder / ScoringEngine 职责分离

### 4. 前端体验不错
- 流式打字效果 + TTS 自动播放
- 面试进行中阻止路由跳转（防误操作）
- 进度条 + 计时器 + 追问标记
- 管理后台功能齐全

## 三、缺点（上线阻断级）

### 🔴 P0 — 安全硬伤

| # | 问题 | 文件 | 严重程度 | 说明 |
|---|------|------|----------|------|
| 1 | 密钥硬编码在 application.yml | `backend/src/main/resources/application.yml` | **致命** | `jwt.secret`、`ai.api-key`、`baidu.speech.api-key/secret-key`、数据库密码全部明文写在配置文件里，Git 可见。上线即泄露 |
| 2 | CORS 全开 | `backend/src/main/java/org/backend/config/WebConfig.java:30` | **高** | `allowedOriginPatterns("*")` 任何域名都能发请求，配合 cookie 凭证可被钓鱼 |
| 3 | 题目 API 免认证 | `backend/src/main/java/org/backend/config/WebConfig.java:44` | **中** | `excludePathPatterns("/api/questions/**")` 所有题目数据无需登录即可获取，竞品可直接爬走整个题库 |
| 4 | AI API Key 无轮换机制 | `backend/src/main/resources/application.yml` | **中** | key 泄露后无法自动失效，无调用频率限制 |

### 🔴 P0 — 数据安全

| # | 问题 | 说明 |
|---|------|------|
| 1 | 无数据库备份方案 | 没有 mysqldump 脚本、没有 binlog 配置、没有定时备份 |
| 2 | 无数据库迁移工具 | 没有 Flyway/Liquibase，表结构靠手动执行 SQL，上线后 schema 变更无法追溯 |
| 3 | 软删除不一致 | `question` 表有 `is_deleted`，但 `interview_session`、`user` 等是物理删除 |
| 4 | 无审计日志 | 只有 AdminLog 记录管理员操作，普通用户的数据操作（创建面试、提交回答）无审计 |

### 🟡 P1 — 架构缺陷

| # | 问题 | 说明 |
|---|------|------|
| 1 | 无 Spring Security | 鉴权靠手写 Interceptor，无法利用方法级权限控制（`@PreAuthorize`）、CSRF 防护、安全头等 |
| 2 | `Map<String, Object>` 入参泛滥 | `InterviewController.create()` 用 Map 接收参数，手动 instanceof，类型不安全，文档不可见 |
| 3 | Controller 手动校验 | create() 里逐个 `if (position == null)`，应该用 DTO + `@Valid` |
| 4 | 单机 Redis 无高可用 | 用的单实例 Redis，无 Sentinel/Cluster，Redis 宕机 = 面试上下文丢失 |
| 5 | AI 调用无重试/熔断 | DeepSeek API 超时或报错时直接失败，无退避重试、无降级方案 |
| 6 | 线程池硬编码 | `evaluateExecutor = newFixedThreadPool(2)` 不能动态调整，高并发时排队 |
| 7 | 无限流 | 没有 Rate Limiting，单用户可无限创建面试、无限调用 AI，账单爆炸 |
| 8 | 前端无错误上报 | 没有 Sentry/前端日志，线上白屏无法定位 |

### 🟡 P1 — 代码质量

| # | 问题 | 说明 |
|---|------|------|
| 1 | 测试覆盖极低 | ~100 个 Java 文件，只有 3 个测试文件，核心 Service 无单元测试 |
| 2 | MyBatis 日志全开 | `log-impl: StdOutImpl` 生产环境会打爆日志 |
| 3 | 无 API 文档 | 没有 Swagger/SpringDoc，前后端对接靠口头约定 |
| 4 | 无健康检查 | 没有 `/actuator/health`，无法做负载均衡健康探测 |
| 5 | 无 Profile 隔离 | 只有一个 `application.yml`，dev/test/prod 共用配置 |
| 6 | 前端无断线恢复 | 面试中断网 = 彻底卡死，SSE 断线已有轮询降级，但可加自动重连 |

## 四、优化执行清单

### 第一阶段：安全加固（预计 1-2 天）

#### 4.1 密钥外部化

**目标：** 所有敏感配置从代码中移除，改用环境变量

**改动文件：**
- `backend/src/main/resources/application.yml`

**步骤：**
1. 创建 `application.yml` 模板，敏感值改为占位符：
   ```yaml
   spring:
     datasource:
       password: ${DB_PASSWORD:}
     data:
       redis:
         password: ${REDIS_PASSWORD:}
   jwt:
     secret: ${JWT_SECRET:}
   ai:
     api-key: ${AI_API_KEY:}
   baidu:
     speech:
       api-key: ${BAIDU_SPEECH_API_KEY:}
       secret-key: ${BAIDU_SPEECH_SECRET_KEY:}
   ```
2. 创建 `application-dev.yml` 放本地开发用的真实值（加入 .gitignore）
3. 创建 `.env.example` 列出所有需要的环境变量名

**验证：** 启动时不设环境变量应报错或用默认空值，设了环境变量后功能正常

---

#### 4.2 CORS 锁定

**目标：** 限制允许的来源域名

**改动文件：**
- `backend/src/main/java/org/backend/config/WebConfig.java`

**步骤：**
```java
// 开发环境
.allowedOriginPatterns("http://localhost:*", "http://127.0.0.1:*")

// 生产环境改为实际域名
.allowedOriginPatterns("https://your-domain.com")
```

**验证：** 从非白名单域名发请求被 CORS 拒绝

---

#### 4.4 题目接口鉴权收紧

**目标：** 题目列表/详情需要登录才能访问

**改动文件：**
- `backend/src/main/java/org/backend/config/WebConfig.java`

**步骤：**
1. 从 `excludePathPatterns` 中移除 `/api/questions/**`
2. 题目详情 API 返回时脱敏：`answer` 字段在列表接口不返回（只在详情接口返回）
3. 保留 `/api/auth/**` 和 `/api/tags/**` 免认证

**验证：** 未登录访问 `/api/questions/list` 返回 401

---

### 第二阶段：稳定性（预计 3-5 天）

#### 4.4 加 Flyway 数据库迁移

**目标：** Schema 版本化管理

**步骤：**
1. pom.xml 引入 `flyway-core`
2. 创建 `db/migration/V1__init_schema.sql`（从现有 seed.sql 提取建表语句）
3. 创建 `V2__add_resume_and_indexes.sql`（resume 表 + 新增索引）
4. application.yml 配置 `spring.flyway.enabled=true`

---

#### 4.5 AI 调用加固

**目标：** AI 调用失败时自动重试，最终降级

**改动文件：**
- `backend/src/main/java/org/backend/util/AIService.java`

**步骤：**
1. 添加重试逻辑：最多 3 次，指数退避（1s → 2s → 4s）
2. 设置合理超时：连接 5s，读取 60s
3. 降级方案：AI 完全不可用时返回预设模板回复
4. 记录每次 AI 调用的耗时和状态（用于监控）

---

#### 4.6 接口限流

**目标：** 防止单用户滥用

**方案：** Redis + Lua 滑动窗口

**限流规则（建议）：**
- 普通接口：60 次/分钟/用户
- AI 面试创建：5 次/小时/用户
- AI 对话发送：20 次/分钟/用户
- 文件上传：10 次/小时/用户

---

#### 4.7 Profile 分离

**目标：** 不同环境使用不同配置

**改动文件：**
- `backend/src/main/resources/application.yml`（公共配置）
- 新建 `application-dev.yml`（本地开发）
- 新建 `application-prod.yml`（生产环境）

**步骤：**
1. 公共配置只放不敏感的通用配置
2. dev 环境用 localhost + 开发密钥
3. prod 环境所有敏感值用 `${ENV_VAR}` 占位
4. 启动时通过 `--spring.profiles.active=prod` 切换

---

#### 4.8 健康检查

**目标：** 可观测应用状态

**步骤：**
1. pom.xml 引入 `spring-boot-starter-actuator`
2. application.yml 配置暴露端点：
   ```yaml
   management:
     endpoints:
       web:
         exposure:
           include: health,info,metrics
     endpoint:
       health:
         show-details: when-authorized
   ```
3. 访问 `/actuator/health` 可检查 DB/Redis 连接状态

---

### 第三阶段：可观测性（预计 2-3 天）

#### 4.9 日志优化

**改动：**
1. application-prod.yml 关闭 MyBatis SQL 日志：
   ```yaml
   mybatis:
     configuration:
       log-impl: org.apache.ibatis.logging.nologging.NoLoggingImpl
   ```
2. 配置 Logback 输出 JSON 格式（便于 ELK 解析）
3. 添加请求日志 Filter：记录每个请求的 URI、耗时、状态码

---

#### 4.10 前端错误监控

**步骤：**
1. 引入 `@sentry/vue`
2. `main.js` 中初始化 Sentry
3. 全局 errorHandler 捕获未处理异常
4. 配置 source map 上传（生产环境）

---

#### 4.11 API 文档

**步骤：**
1. pom.xml 引入 `springdoc-openapi-starter-webmvc-ui`
2. 访问 `/swagger-ui.html` 自动生成文档
3. Controller 方法加 `@Operation` / `@Parameter` 注解补充说明

---

### 第四阶段：功能加强（预计 1-2 周）

#### 4.12 DTO 规范化

**目标：** 所有 Controller 入参用 DTO 替代 Map

**需要改造的接口：**
- `InterviewController.create()` — 创建 `InterviewCreateDTO`（含 `@Valid` 校验）
- `InterviewController.sendMessage()` — 创建 `SendMessageDTO`
- `InterviewController.streamMessage()` — 创建 `StreamMessageDTO`
- 其他使用 `Map<String, Object>` 的接口

---

#### 4.13 测试补全

**优先级排序：**
1. `InterviewService` — 核心业务逻辑，最重要
2. `UserService` — 注册/登录/密码修改
3. `QuestionSelector` — 抽题算法
4. `InterviewEvaluateService` — 评估逻辑
5. `CacheService` — 缓存操作

**目标：** 核心 Service 覆盖率 ≥ 60%

---

#### 4.14 前端体验优化

1. **SSE 自动重连** — 断线后自动尝试重连 3 次，失败再降级轮询
2. **面试中断恢复** — 本地缓存已输入内容，刷新后可恢复
3. **骨架屏** — 列表页/详情页添加加载骨架
4. **错误提示统一** — API 报错时用 ElMessage 统一展示

---

#### 4.15 数据分析埋点

1. 记录用户行为：面试创建、答题、放弃、完成
2. 统计：面试完成率、平均用时、跳出率
3. 题目维度：被选次数、平均得分、追问触发率

## 七、结论

**当前状态：可演示，不可上线。**

核心价值在于功能完整度和 AI 交互体验，在同类项目中明显领先。但安全问题是上线阻断项，不修就是定时炸弹。

**最优先做的 3 件事：**
1. 密钥外部化（1 小时搞定，避免上线即泄露）
2. CORS 锁定 + 题目接口加鉴权
3. AI 调用加固（重试 + 降级）

做完这 3 件事，项目从"课程设计级别"提升到"可小范围内测"水平。
