## 1. 问题概述

### 1.1 一句话总结

**9 个统计分析入口在按用户聚合数据时，未关联面试会话的 position/round，导致跨岗位、跨轮次的不可比数据被混在一起计算。**

### 1.2 问题根因

数据库表设计上，`interview_report` 表**没有冗余存储** position/round，而是通过 `session_id` 关联 `interview_session` 表获取。但 5 个统计 SQL 只查了 `interview_report` 表，没有 JOIN `interview_session`。

```sql
-- 当前（错误）：只按 user_id 过滤
SELECT AVG(total_score) FROM interview_report WHERE user_id = #{userId}

-- 正确：关联 session 表，按 position/round 过滤
SELECT AVG(r.total_score) FROM interview_report r
JOIN interview_session s ON r.session_id = s.id
WHERE r.user_id = #{userId}
  AND s.position = #{position}
```

### 1.3 受影响范围

| # | 功能 | 问题类型 | 严重程度 |
|---|------|----------|:--------:|
| 1 | 仪表盘平均分 | 数值错误 | 🔴 高 |
| 2 | 仪表盘分数趋势图 | 曲线失真 | 🔴 高 |
| 3 | 仪表盘知识雷达图 | 维度分混乱 | 🔴 高 |
| 4 | 仪表盘题目推荐 | 推荐不相关 | 🟡 中 |
| 5 | 仪表盘薄弱标签 | 标签错位 | 🟡 中 |
| 6 | 报告历史列表 | 列表混排 | 🟢 低 |
| 7 | 成长数据 | 进步值错误 | 🔴 高 |
| 8 | 诊断报告历史 | 列表混排 | 🟢 低 |
| 9 | 学习路径 | 学习计划错位 | 🟡 中 |

## 2. 修复策略

### 2.1 核心原则

1. **默认按用户当前目标岗位过滤**
   - 从 `user.target_position` 字段读取
   - 只聚合当前岗位相关的面试数据
   
2. **保留跨岗位查看能力**
   - 前端仪表盘增加岗位筛选下拉框
   - 传 `position=all` 可查看全部（向后兼容）
   
3. **面试轮次的处理**
   - 仪表盘概览不过滤 round（同一岗位的不同轮次都有参考价值）
   - 知识雷达图需按 round 分开（技术面 vs HR面 维度权重不同）
   
4. **后端 API 向后兼容**
   - 新增 `position` 参数，不传时默认用 `user.target_position`
   - 不影响已有前端调用

### 2.2 过滤规则决策

| 场景 | 是否过滤 position | 是否过滤 round | 理由 |
|------|:---:|:---:|------|
| 仪表盘平均分 | ✅ 是 | ❌ 否 | 同岗位不同轮次可合并评估 |
| 分数趋势图 | ✅ 是 | ✅ 是 | 趋势必须在同维度下比较 |
| 知识雷达图 | ✅ 是 | ✅ 是 | HR面和技术面的维度分不可混 |
| 薄弱标签 | ✅ 是 | ❌ 否 | 错题不分轮次 |
| 题目推荐 | ✅ 是 | ❌ 否 | 按岗位推荐题目 |
| 报告列表 | ✅ 是 | ❌ 否 | 列表可混排，用标签区分 |
| 成长数据 | ✅ 是 | ✅ 是 | 同岗位同轮次才可比 |
| 诊断/学习路径 | ✅ 是 | ❌ 否 | 按当前岗位取最新诊断 |

#### 1.4 MistakeMapper — `findCategoryStats` 增加 position 过滤

**错误数据的根本原因**：错题的 category 统计关联的是 question 表的分类，而不是面试的 position。

**修复方案**：统计只计算用户在当前目标岗位面试中产生的错题。

```xml
<!-- 修改：增加 position 关联，只统计指定岗位面试导入的错题 -->
<select id="findCategoryStatsByPosition" resultType="java.util.Map">
    SELECT q.category, COUNT(DISTINCT r.id) AS count
    FROM mistake_record r
    INNER JOIN question q ON r.question_id = q.id AND q.is_deleted = 0
    INNER JOIN mistake_answer_detail d ON r.id = d.mistake_id
    INNER JOIN interview_session s ON d.interview_id = s.id
    WHERE r.user_id = #{userId}
      AND s.position = #{position}
    GROUP BY q.category
    ORDER BY count DESC
</select>
```

> ⚠️ **注意**：`mistake_answer_detail` 表已有 `interview_id` 字段，可以通过它关联到 `interview_session` 获取 position。

#### 2.1 提取"当前岗位"获取逻辑

**核心改动**：新增 `UserService.getCurrentPosition()`，供所有 Service 统一调用

**文件**：`backend/src/main/java/org/backend/service/UserService.java`

```java
/**
 * 获取用户的当前目标岗位
 * 如果未设置，返回 null（调用方自行决定降级策略）
 */
public String getCurrentPosition(Long userId) {
    User user = userMapper.findById(userId);
    if (user == null) return null;
    String position = user.getTargetPosition();
    // 兼容：数据库存的中文名转 code
    if (position != null) {
        return switch (position) {
            case "Java后端开发" -> "java_backend";
            case "前端开发" -> "frontend";
            case "全栈开发" -> "fullstack";
            case "算法工程师" -> "algorithm";
            default -> position; // 已经是 code 则直接返回
        };
    }
    return null;
}
```

## 8. 附录：SQL 变更对照

### 修改前 vs 修改后

| 功能 | 修改前 SQL | 修改后 SQL |
|------|-----------|-----------|
| 平均分 | `WHERE user_id = ?` | `WHERE r.user_id = ? AND s.position = ?` + JOIN session |
| 趋势图 | `WHERE r.user_id = ?` | `WHERE r.user_id = ? AND s.position = ? AND s.round = ?` + JOIN session |
| 知识雷达 | 同上 | 按 round 分开查询，分开展示 |
| 错题统计 | `WHERE r.user_id = ? GROUP BY q.category` | `WHERE r.user_id = ? AND s.position = ?` + JOIN session + JOIN detail |
| 报告列表 | `WHERE r.user_id = ?` | `WHERE r.user_id = ? AND s.position = ?` + JOIN session |
| 最新诊断 | `WHERE user_id = ?` | `WHERE d.user_id = ? AND s.position = ?` + JOIN session |
| 题目推荐 | 无过滤随机取 | 增加 `WHERE direction = ?` |
